# Full-corpus NZ amendment-type classification is feasible from history notes, but needs parser QA before publishing repeal shares

## Executive answer

- **Answer:** A reproducible full-corpus run looks feasible because `jonnonz1/whiplash` already parses PCO XML `<history-note>` blocks into a SQLite amendment graph, and `jonnonz1/nz-statute-book` already renders every public Act consolidation as deterministic git commits from the same PCO XML pipeline [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L1-L23); [nz-statute-book README](https://github.com/jonnonz1/nz-statute-book/blob/1273f588169084689c052a1c685f2a7ec1481720/README.md#L1-L15).
- **History-note shape:** Whiplash documents history notes as element-encoded records containing `<amended-provision>`, free-text `<amending-operation>`, `<amendment-date>`, optional `<amending-provision href="DLM...">`, and `<amending-leg>` title text, then extracts those fields into an `events` table [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L4-L23); [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L210-L225).
- **Operation labels already exist:** Whiplash normalises free-text operations into `inserted`, `substituted`, `replaced`, `repealed`, `revoked`, `amended`, `editorial`, and `other`; the exported churn explorer already separates `amended/inserted/substituted/replaced` from `repealed/revoked` for provision-level events since 2008 [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L50-L70); [whiplash `export_churn.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/export_churn.py#L33-L34); [whiplash `export_churn.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/export_churn.py#L363-L374).
- **Main caveat:** The existing labels are good enough for feasibility, not yet good enough for a final public repeal share, because Whiplash maps raw `omit` to `repealed` while PCO's current drafting guidance says `Delete` is now used instead of `omit` for quoted text, and `Repeal`/`revoke` is used for text that is not in quotation marks [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L50-L60); [PCO amending styles, Wayback snapshot](https://web.archive.org/web/20260201011527/https://pco.govt.nz/instructing-the-pco/drafting-practice-and-model-clauses/new-amending-styles).
- **Practical path:** Run the existing parser against a complete local mirror, add a taxonomy layer that preserves raw operation text plus provision granularity, audit a stratified sample with @jonnonz1, then publish scripts/data tables rather than relying on the rendered git commit bodies alone [whiplash pipeline README](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/README.md#L3-L24); [nz-statute-book README](https://github.com/jonnonz1/nz-statute-book/blob/1273f588169084689c052a1c685f2a7ec1481720/README.md#L26-L41).

**Overall confidence:** Medium — the public code and exported data show the pipeline is real and close to the needed task, but I could not access the multi-GB raw XML mirror or get a response from @jonnonz1 before finishing this issue, and I did not rerun the full mirror locally.

## Evidence

### Scope and collaboration status

This finding answers issue #679's feasibility question, not the stream's final empirical repeal share: it assesses whether PCO history notes, PCO API/XML routes, and the public `jonnonz1` pipeline can support a reproducible full-corpus classification after the seeded sample method is validated [issue #679](https://github.com/thecolab-ai/the-for-good-project/issues/679).

I looped in @jonnonz1 on the issue before writing the finding, stating that I would assess public repo structure, PCO/API documentation, and local history-note/change-operation inspection, and that deeper pipeline-internal claims would remain unverified without public evidence or a response [issue #679 comment](https://github.com/thecolab-ai/the-for-good-project/issues/679#issuecomment-4900343400).

### What the existing pipeline already does

`nz-statute-book` says it contains every public Act of the New Zealand Parliament, every electronic consolidation as a git commit, with 4,001 Acts and 15,282 consolidations current through 2026-06-06 [nz-statute-book README](https://github.com/jonnonz1/nz-statute-book/blob/1273f588169084689c052a1c685f2a7ec1481720/README.md#L1-L9). The same README says the commits are deterministically rendered from PCO XML, with one semantic unit per Markdown line and commit bodies listing amendments effective on that date [nz-statute-book README](https://github.com/jonnonz1/nz-statute-book/blob/1273f588169084689c052a1c685f2a7ec1481720/README.md#L26-L41).

`whiplash` describes its pipeline as `mirror (XML) -> parse history-notes -> amendment graph (SQLite)`, and says the machine-local XML mirror is about 4-6 GB and gitignored; that README self-labels the pipeline page as a placeholder, so I treat the parser and exported data as the stronger evidence that the pipeline exists in practice [whiplash pipeline README](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/README.md#L3-L19). The parser itself keeps only latest versions per work, extracts `<history-note>` tuples, resolves the amending work by `href` first and title/year fallback second, and writes `works`, `events`, and `unresolved` SQLite tables [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L16-L23); [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L200-L259).

The parser normalises free-text operation strings by substring: `insert` to `inserted`, `substitut` to `substituted`, `replac` to `replaced`, `repeal` to `repealed`, `revok` to `revoked`, `amend` to `amended`, `add` to `inserted`, and `omit` to `repealed` [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L50-L70). Export code then groups `amended`, `inserted`, `substituted`, and `replaced` as `CHURN_OPS`, and `repealed` and `revoked` as `DEAD_OPS` [whiplash `export_churn.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/export_churn.py#L33-L34).

The public exported `churn-explorer.json` in the cloned Whiplash repo reports an `ops_note` that amendments are `amended/inserted/substituted/replaced`, repeals are `repealed/revoked`, and the counts are provision-level events since 2008 rather than whole-Act counts; locally reading that file at commit `420d2845d32c38491f3ed603eebedd2ada73572a` gave totals of 131,763 amendments and 25,782 repeals, generated 2026-06-14 [whiplash `churn-explorer.json`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/public/data/churn-explorer.json); [whiplash `export_churn.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/export_churn.py#L363-L382). I treat those totals as evidence of scale and pipeline maturity, not as this stream's final answer, because they group four non-repeal operation types together and depend on parser choices that still need audit.

### Official source access

The PCO developer API documentation says it provides programmatic access to New Zealand legislation and uses a three-tier model of works, versions, and formats [PCO API docs](https://api.legislation.govt.nz/docs/). The same docs say each work may have multiple versions, each consolidation of an Act is a new version, and version formats include HTML, PDF, and XML source URLs that can also be constructed directly from legislation.govt.nz URL patterns [PCO API docs](https://api.legislation.govt.nz/docs/).

For issue #679, the practical implication is that a full-corpus classifier does not have to scrape rendered HTML if it has either a PCO XML mirror or API/XML access; however, the clean developer API route still needs access management because the docs specify per-key rate limits of 10,000 requests per day plus a 2,000 requests per five-minute per-IP burst limit [PCO API docs](https://api.legislation.govt.nz/docs/).

Official-source access should be treated as environment-dependent rather than a substantive blocker. In this worktree on 2026-07-07, `python3 .skills/skills/legislation-nz/scripts/cli.py get-act 1990/109 --version 2022-08-30 --sections-limit 3 --json` returned `upstream_unexpected_html`, `node scripts/fetch.mjs "https://www.legislation.govt.nz/act/public/1990/109/en/2022-08-30.xml/"` returned an AWS WAF JavaScript challenge via plain HTTP, and `node scripts/fetch.mjs "https://catalogue.data.govt.nz/dataset/new-zealand-legislation"` classified the page as `BLOCKED`; however, the PR #688 adversarial review reported that the same URLs were reachable by plain HTTP from its review environment [PR #688](https://github.com/thecolab-ai/the-for-good-project/pull/688); [New Zealand Legislation XML example](https://www.legislation.govt.nz/act/public/1990/109/en/2022-08-30.xml/); [data.govt.nz catalogue page](https://catalogue.data.govt.nz/dataset/new-zealand-legislation). I no longer treat direct XML blocking as a load-bearing reason to prefer a mirror/API; the reason to use a mirror or keyed API is reproducibility, rate-limit management, and avoiding agent-egress variability.

### What needs to change before a full-corpus repeal share is publishable

The main classifier gap is not whether the history notes contain operation text; Whiplash already parses `<amending-operation>` into operation labels [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L8-L13); [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L210-L225). The main classifier gap is that the stream's taxonomy needs legally meaningful distinctions between whole-Act repeal, whole-provision repeal, deletion of words/paragraphs, replacement/substitution, insertion/addition, editorial change, and unresolved/other.

That distinction matters because PCO's current drafting guidance says `Delete` is used instead of `omit`, gives a quoted-word example for deletion, and says `Repeal` or `revoke` is used for text that is not in quotation marks, with a schedule-item example; the same page says `Replace` is used instead of `Substitute` without saying the replaced text is repealed or revoked, and that `Insert` is now used where old drafting split `add` from `insert` [PCO amending styles, Wayback snapshot](https://web.archive.org/web/20260201011527/https://pco.govt.nz/instructing-the-pco/drafting-practice-and-model-clauses/new-amending-styles). The existing Whiplash `omit -> repealed` mapping is therefore a red flag for older history notes: some historical `omit` operations may align with modern repeal/revoke, while quoted-text omissions may align with modern delete, so full-corpus repeal counts should preserve raw operation text and classify both operation wording and target granularity before publishing [whiplash `parse_history.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/parse_history.py#L50-L60); [PCO amending styles, Wayback snapshot](https://web.archive.org/web/20260201011527/https://pco.govt.nz/instructing-the-pco/drafting-practice-and-model-clauses/new-amending-styles).

The second gap is denominator control: `nz-statute-book` is public Acts only and commit-level consolidations, while Whiplash's explorer says its event counts are provision-level events across Acts and regulations since 2008 [nz-statute-book README](https://github.com/jonnonz1/nz-statute-book/blob/1273f588169084689c052a1c685f2a7ec1481720/README.md#L1-L9); [whiplash `export_churn.py`](https://github.com/jonnonz1/whiplash/blob/420d2845d32c38491f3ed603eebedd2ada73572a/pipeline/export_churn.py#L221-L225). A reproducible finding should state whether the frame is public Acts only or Acts plus regulations, whether it starts in 2008 or earlier, and whether it counts history-note events, rendered semantic-line diffs, reprint/consolidation commits, or whole enactments.

## What would change this conclusion

- **A response from @jonnonz1 could raise or lower confidence.** If he confirms that the current graph preserves raw operation text, parser accuracy metrics, unresolved rates, and a complete mirror snapshot, the finding should move toward High confidence; if the public graph is only a prototype or was generated from a partial mirror, it should move toward Low/Medium.
- **Direct access to the raw XML mirror would change the evidence base.** I could not inspect `data-raw/legislation/` because it is intentionally gitignored and multi-GB in the Whiplash repo; a future run with that mirror should report document count, history-note count, unresolved count, operation distribution, and sample audit errors.
- **A direct official XML fetch or API key would reduce access variability.** In this worktree on 2026-07-07, direct legislation.govt.nz XML returned an AWS WAF challenge and the data.govt.nz catalogue page was classified by `scripts/fetch.mjs` as `BLOCKED`, while the PCO API docs were reachable; PR #688 review reported plain-HTTP success for the same URLs from a different environment. A maintained API key, local mirror, or browser-verified bulk XML route would make the official-source path more reproducible.
- **The sample issue could invalidate parser assumptions.** If the seeded sample from issue #676 finds systematic mismatches between history-note operations and legal-text classification, the full-corpus method should be revised before scaling.
- **Legal review is still needed for edge cases.** A full-corpus classifier can label drafting operations, but it cannot alone decide whether a `replace` is substantively a policy repeal, a technical substitution, or a consequential tidy-up.
